### 5.2 Правило нормировки (лист `Параметры`, A30)

> «Порог "без риска" даёт 0 баллов, критический порог — 100 баллов; между ними применяется линейная интерполяция, за пределами — ограничение 0/100.»

Excel-реализация, дословно (строка 2 листа `Расчет_месяц`):

**Направление LOW** (R01, R02, R07) — балл растёт при падении показателя:
```
=MAX(0,MIN(100,(Параметры!$D$5-AT2)/(Параметры!$D$5-Параметры!$E$5)*100))
```

**Направление HIGH** (R03–R06, R08–R15) — балл растёт при росте показателя:
```
=MAX(0,MIN(100,(AU2-Параметры!$D$7)/(Параметры!$E$7-Параметры!$D$7)*100))
```

### 5.4 Итоговый балл — формула дословно (`Расчет_месяц`, BW2)

```
=MAX((BH2*Параметры!$C$5+BI2*Параметры!$C$6+BJ2*Параметры!$C$7+BK2*Параметры!$C$8
     +BL2*Параметры!$C$9+BM2*Параметры!$C$10+BN2*Параметры!$C$11+BO2*Параметры!$C$12
     +BP2*Параметры!$C$13+BQ2*Параметры!$C$14+BR2*Параметры!$C$15+BS2*Параметры!$C$16
     +BT2*Параметры!$C$17+BU2*Параметры!$C$18+BV2*Параметры!$C$19)/100,
     IF(OR(AT2<0.7,AU2<0.6,AD2<0,BG2>=3),75,0))
```

То есть: `Risk Score(r,t) = MAX( Σ wᵢ × Sᵢ / 100 ; критический_override )`, где `Σwᵢ = 100`.

### 5.5 Критический override (жёсткое переопределение)

Формулировка методики (`Методика`, A25):

> «Критический override: итог не может быть ниже 75, если исполнение доходов <70 %, исполнение расходов <60 %, конечный остаток отрицателен либо выявлено ≥3 флагов качества.»

Excel-флаг (`Расчет_месяц`, CE2), дословно:
```
=IF(OR(AT2<0.7,AU2<0.6,AD2<0,BG2>=3),1,0)
```

Реализован как **пол (floor) в 75 баллов**, а не как замена уровня: `MAX(взвешенная сумма; 75)`.

### 5.6 Уровни риска и цвета

### 5.7 Полнота данных (completeness)

`Расчет_месяц`, CC2:
```
=MAX(0,1-MIN(1,BG2/3))
```
Формат ячейки `0%`. То есть полнота = `1 − (число флагов качества / 3)`, ограничена снизу нулём. В отличие от слоя 8.4, **completeness в 8.3 не влияет на уровень риска** — «серого» уровня в этой книге нет.

## 6. Территориальная привязка (слой 8.4)

### 6.1 Механизм

Геопривязка выполняется **по поставщику, а не по заказчику и не по месту поставки**:

```
договор → supplier_bin → registry.«Юр адрес» → Область / Район
```

Обоснование (`Этап 1`, A17 и `Концепция карты`, A5): `delivery_kato` заполнен лишь у 219 из 358 лотов, `customer_bin` — у 61 %; привязка по заказчику давала только 129 из 355 договоров. Привязка по юр. адресу поставщика даёт **355 из 355 (100 %)**.

### 7.3 Формула итогового балла (лист `Формула`, 8 шагов — дословно)

| Шаг | Содержание |
|---|---|
| **1. Нормализация** | «Каждая метрика приводится к безразмерному `v ∈ [0; 1]` через пороги… Если поле пусто… метрика помечается недоступной (`avail = ЛОЖЬ`) и не участвует ни в числителе, ни в знаменателе. **Подстановка 0 запрещена: "нет данных" ≠ "нет риска"**.» |
| **2. Сырая сумма** | `S_raw = Σ (w_i × v_i)` по доступным метрикам i |
| **3. Доступный вес** | `W_avail = Σ (w_i)` по доступным метрикам i |
| **4. Нормированный балл** | `S_norm = 100 × S_raw / W_avail` → 0..100 |
| **5. Коэффициент значимости K** | `K = 1,00 + 0,15 × I_сумма + 0,15 × I_расторгнут`<br>`I_сумма = 1`, если финальная сумма ≥ верхнего квартиля выборки (**8 906 750 тыс. ₸**)<br>`I_расторгнут = 1`, если договор расторгнут<br>Диапазон K: **1,00 … 1,30** |
| **6. Итог** | `Risk Score = min( 100 ; S_norm × K )` → 0..100 |
| **7. Полнота** | `completeness = 100 × W_avail / W_total`, `W_total = 100` |
| **8. Уровень** | см. ниже |

### 7.4 Правило уровня (лист `Формула`, C11 — дословно)

```
если поставщик в Категории A (РНУ или лжепредприятие) → КРИТИЧЕСКИЙ (независимо от баллов)
иначе если completeness < 50% → СЕРЫЙ (ТЗ 7.3)
иначе если Risk Score ≥ 75 → КРИТИЧЕСКИЙ (тёмно-красный)
иначе если ≥ 50 → ВЫСОКИЙ (красный)
иначе если ≥ 25 → СРЕДНИЙ (жёлтый)
иначе → НИЗКИЙ (зелёный)
```

Шкала (`Формула`, C18): `0–24 низкий (зелёный) · 25–49 средний (жёлтый) · 50–74 высокий (красный) · 75–100 критический (тёмно-красный) · полнота < 50 % либо Категория A обрабатываются отдельно`.

Порядок проверок критичен: **Категория A имеет приоритет над серым**.

## 10. Ловушки слоя 8.4

### 10.1 ⚠️ Внутреннее противоречие в трёх «серых» строках

Договоры `10303009`, `10318868`, `10408714`:

- В листе записаны `B1 = 0` (то есть метрика **измерена**), `B2/B3/B4` пусты.
- Тогда `W_avail` должен быть `B1(15)+B5(10)+B6(10)+B7(10)+B8(5)+B9(10) = **60**`.
- Но в листе `W_avail = **45**` — как если бы B1 тоже считался недоступным.
- `S_norm = 100 × 10 / 45 = 22,2` (по листу) против `100 × 10 / 60 = 16,7` (по пересчёту).
- Именно `W_avail = 45 < 50` и делает эти строки «серыми». При корректном `W_avail = 60` они стали бы **«низкий»**.

Причина: у этих договоров `Способ = 'nan'` (строка-заглушка pandas). Генератор пометил B1 недоступным при расчёте `W_avail`, но записал в ячейку `0` вместо пустоты. **Дефект экспорта**: golden-тест должен фиксировать значения листа как есть, но при переносе логики в код это место надо исправить и задокументировать.

## Несогласованности между слоями

1. **Разные шкалы уровней по названию**: 8.3 использует «Низкий/Средний/Высокий/Критический» (с заглавной), 8.4 — «низкий/средний/высокий/критический» (строчные) + «серый (недостаточно данных)». Единый справочник уровней потребует нормализации регистра.
2. **«Серый» уровень есть только в 8.4.** В 8.3 неполнота данных влияет на балл через R15, но не даёт отдельного визуального статуса.
3. **Разная механика override**: в 8.3 это числовой пол (75), в 8.4 — прямая замена уровня. Единая модель карты должна поддерживать оба.
4. **Алматинская область присутствует в обоих слоях**, но в 8.3 как `REG-004` без районов, в 8.4 — с 9 районами и без ID. Для сшивки слоёв нужен общий справочник территорий с `territory_id` и `parent_territory_id`.
5. **Разные пороги «критического»**: в 8.3 он достигается (1 строка = ровно 75,0), в 8.4 — никогда по баллу.
