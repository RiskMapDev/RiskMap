# Слой «Субсидии и господдержка» (ТЗ п.8.5) — план реализации

**Данные:** Excel аналитиков — животноводческие субсидии Алматинской области.
21 521 выплата, 67,5 млрд ₸, 3 413 получателей, 24 названия районов, годы 2021–2025.

**Что реально попадёт в MVP** (проверено на файле):
- сопоставилось с текущей областью — **15 828 строк, 56,73 млрд ₸ (84%), 1 944 получателя**;
- отсеялось — **5 693 строки, 10,8 млрд ₸ (16%)**: районы Жетысу (Панфиловский, Талдыкорган,
  Аксуский, Алакольский, Ескельдинский, Каратальский, Кербулакский, Коксуский, Сарканский,
  Текели), чужие области (Кордайский, Мойынкумский, Ақсуат) и **96 строк с пустым районом**.

Причина: в БД лежат **11 территорий текущей (постреформенной) области**, а файл — по **старой**
области до реформы 2022. Это не баг, это осознанная граница MVP. Всё отсеянное пишется
в `ImportBatch.error_log` с причиной и суммой — аналитик видит, что потерялось.

## 1. Куда что хранится (маппинг на текущую схему)

**Новых таблиц и миграций не требуется.** Схема недели 1 подходит целиком.

| Таблица | Что кладём | Ключ |
|---|---|---|
| `ThematicLayer` | 1 запись: `code='subsidies'`, `name_ru='Субсидии и господдержка'`, `color_hex='#8B5CF6'` | `code` |
| `Territory` | **не трогаем** — уже загружено `load_boundaries` | — |
| `GeoObject` | **1 запись на ПОЛУЧАТЕЛЯ** (не на выплату!) — 1 944 объекта | `(layer, external_id=БИН)` |
| `RiskFactor` | **5 записей на получателя** — расшифровка risk_score (ТЗ п.14) | `geo_object` |
| `ImportBatch` | 1 запись на прогон: 21 521 всего / 15 828 загружено / 5 693 с ошибкой + протокол | — |

### Почему получатель, а не выплата
21 521 выплата в БД не нужна: риск в ТЗ присваивается **субъекту**, вкладка «Риски» показывает
**компании**, карта красит **районы**. Агрегат на получателя = 1 944 строки вместо 21 521 —
быстрее и точно соответствует ТЗ п.14.

### GeoObject — раскладка полей
```
layer         → слой subsidies
territory     → район (сопоставление по нормализованному имени, см. ниже)
external_id   → БИН/ИИН (EnterpriseXin) — ключ идемпотентности
source_system → 'subsidies_xlsx'
imported_at   → время прогона
name          → EnterpriseName
geometry      → Point = point_on_surface(район)   ← координат получателей в файле НЕТ
risk_score    → R, 0–100 (Decimal 5,2)
risk_level    → low | medium | high | critical
attributes    → JSONB, см. ниже
```

**Про geometry честно:** в файле нет ни адресов, ни координат — только название района.
Ставим точку на поверхности района и помечаем `attributes.geo_precision='district_centroid'`
(ТЗ п.15.3 требует маркировать неточную геопривязку). Для MVP это неважно — карта красит
районы, а не рисует точки. Когда подключим реестр юрлиц (неделя 3, слой Арнура) — геокодируем
по адресу регистрации и точки станут настоящими, **без миграции**.

### attributes (JSONB) — сюда уходит всё, для чего нет колонок
```json
{
  "bin": "080340015131",
  "director": "Кабылбеков Нурлан Серикович",
  "paid_total": 11399523080.0,
  "payments_count": 2029,
  "programs_count": 2,
  "programs": ["Заявка на получение субсидий ..."],
  "animal_types": ["Субсидирование в скотоводстве"],
  "share_district": 0.6779,
  "share_oblast": 0.1688,
  "affiliation_cluster_size": 1,
  "anomaly_share": 0.366,
  "outlier_share": 0.286,
  "first_payment": "2021-03-05",
  "last_payment": "2025-06-20",
  "by_year": {
    "2024": {"paid": 2100000000.0, "payments": 412,
             "risk_score": 47.2, "risk_level": "medium"}
  },
  "geo_precision": "district_centroid",
  "district_source_name": "енбекшиказахский"
}
```
`by_year` — это **и есть решение фильтра по периоду** (см. §4).

### RiskFactor — 5 строк на получателя
| indicator_code | вес | raw_value |
|---|--:|---|
| `concentration` | 0.30 | доля в районе (0.6779) |
| `repeat` | 0.15 | число программ |
| `affiliation` | 0.20 | размер кластера по руководителю |
| `process_anomaly` | 0.20 | доля аномальных выплат |
| `amount_outlier` | 0.15 | доля выплат-выбросов |

`contribution = 100 × вес × s` — сумма пяти contribution = risk_score. Это то, что фронт
показывает в карточке как «из чего сложился балл».

### Средний риск района — НЕ храним
В схеме нет поля под риск территории, и **добавлять не надо**. 11 районов × 1 944 объекта —
агрегат считается в SQL мгновенно. Храним риск только у получателя, район агрегируем на лету:

```sql
-- взвешенный по сумме риск района (а не простое среднее!)
SUM(risk_score * (attributes->>'paid_total')::numeric)
  / NULLIF(SUM((attributes->>'paid_total')::numeric), 0)
```
**Почему взвешенный:** простое среднее уравняет ИП с 300 тыс. ₸ и АО с 11 млрд ₸.
В Енбекшиказахском одна «Алель Агро» = 68% денег района — простое среднее это скроет.

## 2. Формула риска (уже реализована в скрипте)

```
R = 100 × (0.30·s1 + 0.15·s2 + 0.20·s3 + 0.20·s4 + 0.15·s5)
```
| | Индикатор | Нормировка в [0;1] |
|---|---|---|
| s1 | Концентрация | `max((доля_в_районе−0.05)/0.45 ; доля_в_области/0.10)` |
| s2 | Повторность | `0.5·(программ−1)/5 + 0.5·(ln(выплат)−ln3)/(ln100−ln3)` |
| s3 | Аффилированность | `(получателей_у_руководителя−1)/2` |
| s4 | Процессные аномалии | `доля_аномальных_выплат / 0.30` |
| s5 | Выбросы сумм | `доля_выплат-выбросов / 0.20` |

**Уровни (ТЗ п.7.3):** `low` 0–34 · `medium` 35–54 · `high` 55–74 · `critical` 75–100 · нет данных → серый.

**Факт на данных:** low 1883, medium 59, high 2, critical 0. Топ: ИП Тогузбаев R=72.1,
СПК RAIYMBEK QAZYNA R=64.9, АО «Алель Агро» R=50.6 (11,4 млрд ₸).

> ⚠️ Пороги должны совпасть с общим стандартом уровней от Маржан. Если стандарт другой —
> правится одна константа `RISK_BANDS` в скрипте, пересчёт = повторный прогон.

**Почему «Алель Агро» только medium, хотя это 17% всей области:** R — это оценка
*поведения*, а не размера. Крупный, но процедурно чистый получатель = средний риск.
Чтобы не потерять материальность, на дашборде показываем **риск-экспозицию = сумма × R/100**:
у «Алель Агро» она 5,77 млрд ₸ — первое место в области. Обе метрики нужны, они отвечают
на разные вопросы: «кто подозрителен» и «где лежат деньги под риском».

## 3. Скрипт импорта — готов

`backend/territories/management/commands/import_subsidies.py`

```bash
pip install pandas openpyxl msoffcrypto-tool      # добавить в requirements.txt
python manage.py import_subsidies --file subs.xlsx --password 0101 --dry-run
python manage.py import_subsidies --file subs.xlsx --password 0101 --user marzhan
```
- `--dry-run` — считает и печатает топ-10, в БД не пишет;
- `--password` — файл аналитиков зашифрован (agile encryption), снимаем в память;
- идемпотентен: повторный прогон обновляет по `(layer, БИН)`, факторы пересоздаёт;
- всё в одной транзакции: упало — не осталось наполовину загруженного состояния.

## 4. Фильтры: что реально сделаем, а что нет

### Уровень риска — полноценно
`?risk_level=high,critical` фильтрует **объекты**, район пересчитывается по отфильтрованному
множеству. Район без подходящих объектов → серый («Нет данных»). Одна логика и для заливки
районов, и для списка во вкладке «Риски» — не разъедется.

### Период — по годам (честное ограничение MVP)
На макете уже есть кнопки **2022 / 2023 / 2024** — это и делаем: `?year=2024` читает
`attributes.by_year['2024']` (там своя сумма И свой risk_score — риск пересчитан на срезе года).

**Произвольный диапазон дат (01/01/2024–30/06/2024) в MVP не поддерживаем.** Причина честная:
чтобы пересчитать риск на произвольном интервале, нужны сырые выплаты в БД — а мы храним
агрегат на получателя. Варианты:
- **MVP:** поля дат в фильтре скрыть/задизейблить, оставить кнопки годов;
- **Фаза 2:** таблица `SubsidyPayment` (21 521 строка) + пересчёт на лету. Это новая
  миграция — решать отдельно, не тащить в MVP.

Не делайте вид, что диапазон работает, фильтруя только деньги: получится, что сумма
изменилась, а риск нет — аналитик решит, что система врёт.

## 7. Порядок и зависимости

```
Нурасыл: импорт (д.1) → /territories/risk/ (д.2) → /dashboard/ (д.3) → /geo-objects/ (д.4)
                             ↓                          ↓                    ↓
Нурай:   карта на моках (д.1-2) → заливка (д.2) → зум+дашборд (д.3) → «Риски» (д.4)
Мухаммед: фильтры на моках (д.1-2) ────────────→ подключение (д.3) → импорт-мастер (д.4)
```
**Критический путь — Нурасыл.** Поэтому: контракт API (§5) фиксируем **в понедельник утром**,
до написания кода, и оба фулстека сразу работают на моках по нему. Тогда день 3 — это
подключение, а не переписывание.

## 8. Договорённости, которые снимут конфликты

1. **Цвета — только на фронте**, в одном конфиге. Бэк отдаёт `risk_level` строкой.
2. **`risk_level` — единый словарь** `low|medium|high|critical` во всех слоях (не «Высокий»/«high»/«HIGH»).
3. **Фильтры живут в URL** — иначе карта и «Риски» разъедутся.
4. **Никаких новых миграций** под этот слой. Не хватает поля → в `attributes`.
5. **Средний риск района — только взвешенный по сумме.** Простое среднее в код не попадает.
6. Прогон импорта = один источник цифр. Спорите о числе — смотрите `ImportBatch`, не Excel.
