# Источник границ

Данные — из **OpenStreetMap**, актуальные на момент загрузки (в отличие от
первой версии на GADM 3.6, которую заменили: geoBoundaries.org и GADM
оказались снимками 2017-2019 гг. и не отражали реформу 2022 года).

## Как получены файлы

1. **Список регионов и районов** — через Overpass API:
   ```
   relation["boundary"="administrative"]["admin_level"="4"]["ISO3166-2"~"^KZ-"];
   ```
   дало все 20 текущих регионов (17 областей + Астана, Алматы, Шымкент —
   включая Абайскую, Жетысускую, Улутаускую области, добавленные в 2022 г.).

   Районы конкретного региона — через `area(36000000000+relation_id)` +
   `admin_level=4`. Запрос по площади цепляет объекты из соседних стран
   у границы (Кыргызстан, Китай) и городские районы самого города рядом —
   их отфильтровали вручную по тегу `addr:country=KZ` и здравому смыслу
   (районы г. Алматы — не районы области, хоть и соседствуют).

2. **Геометрия каждого relation** — через сервис
   `polygons.openstreetmap.fr/get_geojson.py?id=<relation_id>` (он же
   используется multiple GIS-проектами для выгрузки OSM-мультиполигонов
   в готовый GeoJSON — вручную собирать relation из ways ненадёжно).

3. Итоговые файлы:
   - `kz_regions_osm.geojson` — 20 регионов. `properties`: `osm_relation_id`,
     `iso3166_2`, `kato_code`, `name_ru`, `name_kk`.
   - `kz_almaty_districts_osm.geojson` — 11 единиц Алматинской области
     (9 районов + 2 города обл. значения: Қонаев, Алатау) в **текущих**,
     постреформенных границах — меньше исходных 17, часть ушла в Жетысу.

## КАТО-коды

- **Регионы**: код = `<номер ISO3166-2> + "0000000"`, напр. `KZ-19` →
  `190000000`. Формула проверена сверкой с tenderplus.kz/kato — совпадает
  (Алматы `750000000`, Астана `710000000`, Шымкент `790000000` и т.д.).
- **Районы**: КАТО намеренно **не проставлены**. У tenderplus.kz ветка
  Алматинской области (190000000) на уровне районов ещё показывает старый
  состав (Аксу, Алаколь, Каратал и др. — уже переданные в Жетысу), то есть
  сама справочная база на leaf-уровне не синхронизирована с реформой.
  Подставлять код наугад хуже честного пропуска.

## Известные огрехи автосборки

4 из 20 регионов пришли с невалидной топологией («nested shells» —
типично для областей с городами-анклавами внутри). Чинится в
`load_boundaries.py` через `geometry.make_valid()` (GEOS ≥ 3.10)
при каждой загрузке — руками координаты не правили.

## Обновление данных

Если появится официальный источник (геопортал РК, статкомитет) с более
точной геометрией/КАТО — заменить эти два файла и перезапустить
`python manage.py load_boundaries`, код команды менять не нужно.
