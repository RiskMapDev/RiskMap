# Неделя 1 и дорожная карта на 5 недель — «Интерактивная карта рисков»
Команда: Маржан (тимлид + аналитик), Арнур, Камилла, Асия (аналитики), Мухаммед, Нурай (full-stack), Нурасыл (backend)

---

## Дорожная карта: кто какой слой ведёт и когда он реализуется

С 4 аналитиками логика меняется: **все четверо исследуют свои слои параллельно уже на неделе 1**, пока три разработчика делают базовый Административно-территориальный слой. Дальше каждую неделю разработчики реализуют один готовый слой — очередь по мере готовности спецификаций, не по мере исследования (исследование уже закончено у всех к пятнице недели 1).

| Неделя | Слой | Кто вёл исследование | Источники |
|---|---|---|---|
| 1 | Административно-территориальный (база) | Мухаммед / Нурай / Нурасыл строят, все 4 аналитика параллельно исследуют свои слои | geoBoundaries.org |
| 2 | Государственные закупки | **Маржан** | goszakup.gov.kz API |
| 3 | Хозяйствующие субъекты | **Арнур** | goszakup /subject, portal.kgd.gov.kz, elicense.kz |
| 4 | Инфраструктурные проекты и ГЧП | **Камилла** | epsd.kz/qportal.kz, ppp.ecc.kz/kzppp.kz |
| 5 | Бюджетный | **Асия** | budget.egov.kz, openbudget.kz |

Итого 5 слоёв за 5 недель — админ-территориальный как база плюс 4 тематических, ровно по числу аналитиков. Социально-экономические показатели (население, доходы, безработица со stat.gov.kz) не выделяем в отдельный слой — заводим как дополнительные атрибуты территории уже на неделе 1, это дешевле, чем держать под них пятого аналитика.

**Важно про масштаб карты (по твоей вводной):** на карте отображается весь Казахстан на уровне областей (чтобы система выглядела универсальной), но детализация до районов и реальные данные — только по Алматинской области. Это сразу закладываем в архитектуру недели 1, чтобы потом не пришлось ничего переделывать, просто дозагружать районы других областей позже.

**Как работают аналитики на неделях 2-5, пока не их слой в разработке:** не простаивают — валидируют качество данных слоя, который реализуется на этой неделе (второй взгляд полезен всегда), донастраивают веса своего уже реализованного или ещё не реализованного слоя, готовят чистый файл данных к своей неделе заранее.

## Неделя 1 — подробный план по каждому

### Нурасыл (backend) — ядро БД и API

- **Пн:** Репозиторий, docker-compose (PostgreSQL+PostGIS, Django, Redis-заглушка). Django-проект, GeoDjango подключён. Проверить, что весь стек поднимается одной командой у всех на ноутах.
- **Вт:** Модели `Territory` (kato_code, name, level, parent_id, geometry, population), `ThematicLayer` (реестр слоёв), `GeoObject` (универсальная таблица под будущие слои 2-5), `RiskFactor`, `ImportBatch`. Миграции. Django admin.
- **Ср:** Management-команда загрузки границ: 
  — все области Казахстана (ADM1 из geoBoundaries) — только контур, без районов;
  — районы (ADM2) — только для Алматинской области.
  Технически сложный момент: связь "район → область" через `ST_Contains` в PostGIS (центроид района внутри полигона области), не через сопоставление имён.
- **Чт:** DRF-эндпоинты: `GET /territories/` (с фильтром по уровню — область/район), `GET /territories/{id}/`. Синхронизация формата ответа с Нураем.
- **Пт:** Помощь Мухаммеду с backend-частью импорт-мастера, код-ревью, README для запуска окружения.

**Итог у Нурасыла:** рабочие миграции, API территорий (вся страна на уровне областей + Алматинская область детально), воспроизводимое окружение.

### Нурай (full-stack) — карта и карточки

- **Пн:** React+TS (Vite), базовый layout по Figma-макету (сайдбар, шапка).
- **Вт:** Leaflet подключён, рендер границ всех областей Казахстана (пока на моках, если API ещё не готово).
- **Ср:** Карта подключена к реальному `/territories/`, области кликабельны, при клике на Алматинскую область — зум и подгрузка районов. Панель слоёв — пока один пункт, но собранная так, чтобы список подгружался динамически (место под слои 2-5).
- **Чт:** Клик по территории → боковая карточка (название, уровень, население, площадь), подключена к API.
- **Пт:** Полировка, интеграционное тестирование с реальными данными Нурасыла.

**Итог у Нурая:** интерактивная карта Казахстана с детализацией по Алматинской области и рабочей карточкой территории.

### Мухаммед (full-stack) — универсальный импорт-мастер

- **Пн:** Знакомство с репозиторием и схемой вместе с Нурасылом (парное программирование по моделям `ImportBatch`/`GeoObject`).
- **Вт:** Backend импорт-мастера — загрузка Excel/CSV, парсинг заголовков, эндпоинт возврата колонок для маппинга.
- **Ср:** Frontend импорт-мастера — 3 шага (загрузка → маппинг колонок → предпросмотр/валидация с подсветкой ошибок).
- **Чт:** Полный тест на реальном файле — просит у любого аналитика Excel с населением по районам Алматинской области (со stat.gov.kz) и наполняет `Territory.population` через свой же импорт-мастер.
- **Пт:** Правки по итогам теста, короткая инструкция "как пользоваться импорт-мастером" для аналитиков — им предстоит пользоваться этим постоянно на неделях 2-5.

**Итог у Мухаммеда:** рабочий, проверенный на живых данных импорт-мастер — им будут заводиться все 4 тематических слоя дальше.

### Маржан (тимлид + аналитик, слой — Государственные закупки)

**Организационное:**
- Пн, утро: письмо на токен goszakup API в АО «ЦЭФ» — первым делом, это на бюрократии и блокирует твою же реализацию на неделе 2.
- Пн: кик-офф — разбор этого плана с командой из 7 человек, явно проговорить зону ответственности каждого аналитика.
- Каждый день: короткий standup (15 минут) — у 7 человек особенно важно не дать кому-то тихо застрять.
- Ср: сверка прогресса и по разработке, и по всем 4 аналитическим направлениям параллельно — не только своим.
- Пт: общее мини-демо (карта + карточка + импорт-мастер) + короткая сверка готовности всех 4 спецификаций слоёв к следующим неделям. Резюме недели — в группу/руководителю.

**Аналитика (слой госзакупок):**
- Data dictionary по goszakup API (subject, trd-buy, contract, rnu).
- JSON-схема для `GeoObject.attributes` под закупки.
- Индикаторы риска и их точная логика расчёта (РНУ, единственный источник, демпинг, повторные победы у одного заказчика).
- Черновые веса, сверенные на 2-3 реальных примерах.
- Формализация уровней риска (зелёный/жёлтый/красный/бордовый — конкретные диапазоны баллов) — это нужно всем 4 слоям как общий стандарт, имеет смысл предложить его именно тебе, раз твой слой идёт первым в реализации на неделе 2.
- UI/UX-набросок: как объект закупки выглядит на карте (иконка/цвет маркера) и что показывает карточка — можно как правки поверх уже готового Figma-макета.
- Итог пятницы: "Спецификация слоя закупок v1", полностью готовая к имплементации в понедельник недели 2.

### Арнур — аналитик, слой Хозяйствующие субъекты

- Изучить структуру goszakup `/subject` + portal.kgd.gov.kz (неблагонадёжные налогоплательщики, сведения по контрагентам) + elicense.kz.
- Data dictionary + JSON-схема для `GeoObject.attributes` слоя организаций.
- Индикаторы риска: флаг РНУ, флаг из списка неблагонадёжных КГД, статус лицензии.
- Черновые веса.
- UI/UX-набросок: как организация отображается на карте (это точечный объект или скорее привязка к адресу регистрации?) и что содержит карточка организации.
- Итог пятницы: "Спецификация слоя организаций v1" — готова к неделе 3.

### Камилла — аналитик, слой Инфраструктурные проекты и ГЧП

- Изучить структуру epsd.kz/qportal.kz (архив ТЭО/ПСД) + ppp.ecc.kz/kzppp.kz (реестр договоров ГЧП, реестр недобросовестных частных партнёров).
- Data dictionary + JSON-схема для слоя проектов.
- Индикаторы риска: в реестре недобросовестных партнёров, проект без завершённой экспертизы.
- Черновые веса.
- UI/UX-набросок: карточка проекта (участники, сроки, сумма, статус).
- Итог пятницы: "Спецификация слоя инфраструктуры и ГЧП v1" — готова к неделе 4.

### Асия — аналитик, слой Бюджетный

- Изучить budget.egov.kz и openbudget.kz — отчёты об исполнении бюджета по областям (учти: данные там документами Word/Excel/PDF, а не через API — оцени реальный объём ручной работы по своду в таблицу).
- Сузить до конкретных отчётов по Алматинской области за последние 1-2 года.
- Индикаторы риска: % освоения бюджета, отклонение от плана по программам.
- Черновые веса.
- UI/UX-набросок: у бюджетного слоя нет точечных объектов как у закупок — вероятно, это заливка территорий цветом по % освоения (хороплет), а не маркеры. Продумать это отдельно, это другая логика отображения.
- Итог пятницы: "Спецификация бюджетного слоя v1" + черновая сведённая таблица данных по Алматинской области — готова к неделе 5.

## Что должно получиться у всей команды к вечеру пятницы

- Любой поднимает весь стек одной командой на своём ноутбуке.
- На карте — все области Казахстана контуром, Алматинская область — с районами и реальным населением, загруженным через собственный импорт-мастер.
- Клик по территории открывает карточку с реальными данными.
- Импорт-мастер протестирован на живом файле от начала до конца.
- 4 готовые, детальные спецификации тематических слоёв (закупки, организации, инфраструктура/ГЧП, бюджет) — недели 2-5 стартуют с имплементации, а не с исследования.
- Единый стандарт уровней риска (пороги для зелёного/жёлтого/красного/бордового), общий для всех 4 слоёв.
- Токен на goszakup API запрошен.

Если что-то из этого не готово к пятнице — сигнал притормозить и разобраться, а не тянуть проблему на неделю 2.
